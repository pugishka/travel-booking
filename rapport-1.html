Les contraintes OCL :


  1. Un aéroport est identifié par trois lettres uniques à chaque aéroport.
    2. 
context Aéroport inv:
    self.allInstances() -> forAll(c1, c2:
    | c1 <> c2 implies c1.idAéroport <> c2.idAéroport)
  2. La partie alphabétique de l'ID d'un vol est unique à chaque compagnie et la partie numérique est unique à chaque vol au sein de la même compagnie.
    3. 
context Compagnie aérienne
inv: let 
            2_prem_chars = self.Vol.idVol.at(0) + self.Vol.idVol.at(1)
            last_chars  = self.Vol.idVol.excludes(2_prem_chars)
        in
            self.allInstances() -> forAll(c1, c2:  
            | c1 <> c2 implies c1.first2chars <> c2.first2chars)
            self.Vol.allInstances() -> forAll(v1, v2: Vol 
            | v1 <> v2 implies v1.last_chars <> v2.last_chars)
  3. L'aéroport de départ et d'arrivée d'un vol doit être différent.

    4. 
context Aéroport
   inv: self.Vol.origine <> self.Vol.destination
  4. Tous les sièges d'une même section ont le même prix.

    5. 
context Siege inv:  
    Siege.allInstances() -> forAll(s1, s2| s1.section = s2.section implies s1.prix = s2.prix )
  5. Un port est identifié par trois lettres uniques à chaque port.

    6. 
context Port inv:
    self.allInstances() -> forAll(p1, p2:
               | p1 <> p2 implies p1.idPort <> p2.idPort)
    self.idPort.size() = 3
  6. Un itinéraire ne peut pas durer plus de 21 jours.

    7. 
context Itineraire 
    inv: self.dateArrive - self.dateDepart < 22
  7. Chaque itinéraire visite une suite de ports fixes. Le port de départ et d'arrivée doit être le même.

    8. 
context Itineraire
    inv: self.port.first() = self.port.last()
  8. Un paquebot peut être assigné à plusieurs itinéraires tant qu'ils ne se chevauchent pas.

    9. 
context Paquebot
    inv: self.itinéraire.allInstances() -> forAll(p1, p2:
		| p1 <> p2 implies p1.dateDepart > p2.dateArrivee or p2.dateDepart > p1.dateArrivee
  9. Toutes les cabines d'une même section ont le même prix.

    10. 
context Compagnie de croisière
   inv: self.itineraire.cabine.allInstances() 
      -> forAll(s1, s2: self.itineraire.cabine
         | s1 <> s1 implies s1.prix = s2.prix)
  10. Une gare est identifiée par trois lettres uniques à chaque gare.

    11. 
context Gare inv:
    self.allInstances() -> forAll(g1, g2:
               | g1 <> g2 implies g1.idGare <> g2.idGare)
    context Gare
    inv: self.idGare.size() = 3
  11. Le client peut réserver un siège disponible dans un vol (ligne) donné.

    12. 
context Siege inv: self.confirmation = null and self.reservation = null
  12. Un siège réservé devient assigné à un passager une fois payé: le siège est donc confirmé.

  13. Le client peut réserver une cabine disponible pour un itinéraire donné.

    14. 
context Cabine inv: self.confirmation <> null and self.reservation <> null

Tâche 4


L’utilisateur commence par entrer son nom et mot de passe afin de se connecter à partir du menu principal contenu dans la classe “Main”. Un message d’erreur lui est envoyé dans le cas ou il y a une erreur avec ses informations ou bien que le email n’est pas associé à un compte utilisateur. Dans le cas ou il s’agit d’un client, le menu client s’ouvre, sinon le menu administrateur s’ouvre. Le menu client permet d’accéder au menu voyage qui donne accès aux méthodes des trois différentes classes de voyage qui sont en lien avec la consultation des voyages disponibles, la réservation de siège / cabine et l’achat. Dans le cas d’un utilisateur admin, il a maintenant accès à toutes les méthodes disponibles dans les classes de voyage, notamment la création de nouveaux voyages, compagnie ou destination. Il nous semblait important de mettre les deux types de menu en un, mais avec les méthodes admin cachées ou non en fonction de l’utilisateur afin d’éviter la duplication de code pour les méthodes qui sont partagées par les deux types d’utilisateur.

Dans le cas où l'utilisateur fait appel aux méthodes réserver() et acheter(), il se fait renvoyer une réservation et passe au menu de confirmation qui contient la méthode payer(), suite à laquelle l’utilisateur reçoit une confirmation de paiement et du siège.

Nous avons choisi de mettre toutes les méthodes liées à un type de voyage dans la classe de voyage lui étant associé afin de réduire au maximum le couplage et d’augmenter la cohésion.
